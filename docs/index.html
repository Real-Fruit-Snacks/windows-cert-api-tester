❯ THE PROBLEM

Client-certificate endpoints, without the file wrangling

Some internal sites don't take a password — they ask your browser to choose a certificate and complete a mutual-TLS handshake. Most API clients want that certificate and its private key as files on disk, which enterprise and smart-card certificates deliberately don't allow. Certificate API Tester pulls the certificate straight from your Windows store and lets Windows sign the handshake, so the private key never has to leave — and never has to be exportable. The certificate is optional, so it also works as a general-purpose API client.

❯ WHAT IT DOES

Features

Windows store certs

Pick a client certificate from CurrentUser\My with subject, issuer, thumbprint, expiry, and a filter box. Non-exportable and smart-card keys work — Windows signs the handshake.

Certificate optional

Leave it on “no certificate” to send an ordinary request. It doubles as a general API tester for endpoints that don't require mutual TLS.

Full request builder

Any method, a key-value query-parameter grid and headers grid, Bearer/Basic auth helpers, a body with a content-type selector, a timeout, and a Cancel button.

Request tabs

Keep several requests open at once, each with its own website, parameters, headers, body, auth, certificate, and response. New with + or Ctrl+T; your open tabs return next launch.

Query parameters

A dedicated Params tab with an enable/key/value grid. Paste a URL with a ?query and it splits into the grid; it recombines, correctly encoded, when you send.

Collections

Save named requests into folders and reopen them in a tab. A History / Collections switch in the sidebar, with save, folder, rename, and delete. Persists between sessions.

Known-good endpoints

Every saved request remembers its last result — a mint dot for a 2xx, red for a failure, with a tooltip showing when it was last checked and what it returned. See at a glance which endpoints are verified working.

Environments & variables

Define {{variable}} values per environment and switch from the ENV selector. Substituted in URL, query, headers, body, and auth when you send; unresolved tokens are flagged.

Capture & reuse auth tokens

Grab a value from a response — a JSON field like access_token or a header — into a {{variable}}. Call your auth endpoint once, then send Authorization: Bearer {{token}} on every later request. In the app (a Capture tab) and headless (certapi send --capture) — or skip the setup entirely with automatic detection.

Automatic tokens

Log in once — the token in the response is detected, scoped to that host, and attached to follow-on requests automatically. GUI, CLI, and MCP server alike.

Import from cURL

Paste a curl command and it opens a ready-to-send tab — method, URL, query, headers, body, and auth all filled in. Understands the common flags, quoting, and line continuations.

Import OpenAPI

Point it at a JSON OpenAPI 3.x or Swagger 2.0 file to generate a collection of requests, foldered by tag, with the server as each request's website.

Export as OpenAPI

Write a folder — or all collections — as an OpenAPI 3.0 JSON file: folders become tags, requests become operations with parameters and body examples, and known-good notes become descriptions. Auth is exported as a scheme only — never the secrets.

Pop-out response views

Open a single response view or the whole response panel — tabs and all — in its own window. Everything stays live, including the Network trace and Rendered pages, and closing the window puts the content back.

Save / load workspaces

Export everything — tabs, collections with known-good results, environments, saved websites, history — to one JSON file and load it back later, merging or replacing. Move machines, keep project snapshots, or share a ready-to-use setup.

Headless mode

A separate certapi.exe drives everything from scripts: one-off mTLS requests, saved requests and collections as pass/fail suites, certificate listing, self-test, and import/export — body to stdout, diagnostics to stderr, meaningful exit codes.

Local mTLS gateway

certapi serve runs a loopback reverse proxy to a certificate-protected site: point any local app's base URL at http://localhost:<port> and it reaches the upstream with your client certificate attached — no mTLS code of its own. Loopback-only, optional token.

MCP server for AI agents

certapi mcp exposes certapi over the Model Context Protocol: an AI agent can send mTLS requests and run saved ones, using a certificate you pin at launch and bounded by a host allowlist — the agent never handles certificates itself. JSON-RPC over stdio, nothing on the network.

Rendered website view

A Rendered tab opens the URL as a web page and fetches every resource — document, CSS, JS, images, XHR — with your client certificate, so a cert-protected site renders fully. Loads on demand via the Windows WebView2 runtime.

Built-in help

A ? in the title bar (or F1) opens a Help & Reference window covering every feature, the keyboard shortcuts, and an About panel — all embedded, so it works with no web access.

Reads any response

Pretty-prints JSON and XML with syntax highlighting, shows HTML/text, hex-dumps binary, and sniffs the body when the content type is missing or lying.

Connection diagnostics

See the negotiated TLS version and cipher, whether your client cert was actually presented, and the server's certificate and chain.

Network trace

A Network tab like a browser's network panel — every HTTP call logged, including every resource the Rendered view fetches, with method, status, type, size, timing, and a client-certificate marker. Filter by text, status class, or cert-only; click a row for a resizable details pane with headers; right-click to copy the URL or a curl command.

Saved websites

Save a base URL like https://internal.corp and the URL box becomes just the path after it — fire off /api/thing without retyping the host.

Request history

A sidebar of recent requests, labelled by path. Click one to reload the entire request — website, cert, headers, auth — and the response it returned.

Copy & export

Copy the response body, copy the request as a cURL command, and save any response with a sensible file extension.

Clear failures

Tells “certificate refused” apart from “server not trusted”, a network error, and a timeout — with an off-by-default insecure bypass for private CAs.

Honors your proxy

Follows the machine's proxy, including WPAD auto-detect and a PAC script from Internet Options, authenticating with your Windows credentials.

Built-in self-test

One click stands up a local mutual-TLS server and proves the whole certificate path end to end — no real endpoint required.

❯ GET IT

Download or build

Download

Grab ApiTester.App.exe from the latest release and double-click it. No install, no admin — copy it anywhere and run it. The command-line client certapi.exe is a separate asset on the same releases page.

Build from source

dotnet build
dotnet test --filter "Category!=StoreRoundTrip"
dotnet run --project src/ApiTester.App

# portable exes (app + command line):
dotnet publish src/ApiTester.App -c Release \
  -r win-x64 --self-contained -o publish
dotnet publish src/ApiTester.Cli -c Release \
  -r win-x64 --self-contained -o publish

# see exactly what a command did, on screen or in a file:
certapi send https://api.example.com/x --debug
certapi run --all --log-file certapi.log

❯ NO DEPENDENCIES

A single self-contained executable

The released executable bundles the .NET runtime, so running it is just copying one file to any Windows 10/11 machine — no installer, no admin rights, and nothing to install first. The one optional exception is the Rendered website view, which uses the Microsoft Edge WebView2 runtime that ships with Windows 11 (and is a standard component on up-to-date Windows 10); it loads only when you open that tab, and everything else works without it. To build it on your own infrastructure, the repository ships a .gitlab-ci.yml for a self-hosted GitLab that builds, tests, and packages the executable on a Windows runner and can publish this documentation site to GitLab Pages.

❯ UNDER THE HOOD

How it works

Mutual TLS

The app builds an HttpClient over a SocketsHttpHandler and attaches your chosen certificate. During the handshake the server requests a client certificate and the app presents yours; for non-exportable keys the signing is done by Windows, so the app never sees the raw key.

Browsing with your cert

The Rendered tab hosts a WebView2 and intercepts every resource the page requests — document, styles, scripts, images, XHR — fetching each through the same client-certificate HttpClient. So a certificate-protected page renders fully authenticated, not just its first response.

Live diagnostics

For direct connections the app performs the TLS handshake itself, so it can report the negotiated protocol and cipher and whether your certificate was actually presented. The server certificate and chain are always captured.

A real self-test

The self-test generates an in-memory CA plus a server and client certificate, runs a local server that requires a client certificate, and drives a real request through the same code path used for live endpoints.

Your proxy, your rules

Requests follow the machine's configured proxy — WPAD auto-detect and a PAC script from Internet Options — and authenticate to it with your Windows credentials when the proxy requires it.
